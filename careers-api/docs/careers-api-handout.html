Careers API: Job Ads Integration — One-Page Reference

HR approves in the Dynamics job portal
→
Dynamics pushes the ad one secure call to WordPress

The flow at a glance

One approved job's journey from Dynamics to the public careers page. Details for each numbered step follow below.

The endpoint

https://<site>/wp-json/careers-api/v1/jobs/{job-id}
{job-id} is Dynamics' own job reference (e.g. JOB-2026-014). It must never change for a given job — WordPress stores it on the ad (meta field _careers_api_job_id) and matches every later message by it. WordPress internals are never exposed.
PUT

Create or update in one verb. Send the job's current state; WordPress makes the site match. Unknown ID → created & published (201). Known ID → overwritten (200). Re-sends are always safe — never duplicates.

Closing a vacancy = PUT with a past deadline. The ad moves to the "Closed" tab automatically and stays visible as history.

DELETE

Retract a mistake. Removes the ad from the site entirely (recoverable from trash). 404 if the ID is unknown.

Never used for routine closing — only for "this should not have been posted."

The JSON contract (PUT body)

FieldRequiredFormatNotes
titleyestextthe job title shown everywhere
descriptionyestext / HTMLthe full advert, shown on the job's own page. Normal formatting (paragraphs, lists, links) survives; WordPress auto-removes anything that could run as code (scripts, embeds) before saving — Dynamics doesn't need to clean it
short_descriptionnotext / HTMLthe short teaser under the title on the careers listing card. Optional — if not sent, the card just shows title, deadline & location. Same auto-cleaning applies
locationyes2-letter ISO codeKE, not "Kenya"
application_deadlineyesYYYY-MM-DDdate only — convert from OData timestamp, mind the timezone
Field-name rule: WordPress defines this contract; Dynamics maps its own OData field names into it when building the message. Internal renames on the Dynamics side only ever touch that one mapping. Invalid input is rejected whole (400, named field) — nothing half-saves.

Authentication

StepWhoWhat
1WP adminCreate user with role Job Portal Integration (can write job ads only — nothing else on the site)
2WP adminThat user's Profile → Application Passwords → generate one ("Dynamics Job Portal"); shown once, revocable in one click
3DynamicsStore site URL + username + password in a secrets vault (never in source); send via HTTP Basic auth over HTTPS (WP refuses app passwords on plain HTTP)
No extra plugin needed: all of this uses WordPress's normal, built-in user screens — same as managing admins and editors. The account is an ordinary user; the "Job Portal Integration" role is registered automatically when the Careers API plugin is activated and then appears in the standard role dropdown; Application Passwords are core WordPress (since 5.6), on every user's profile page — the section only shows once the site is on HTTPS.

What WordPress does on receipt

A DELETE runs the same checks (1–3 and 5), then moves the ad to the trash — 404 if the job ID is unknown.

Reliability rules for the sender

Send only on change (approve / edit / retract) — no scheduled bulk re-sync. Retry 5xx / network failures with backoff — 5xx = server-side errors, 500–599 (e.g. 500 Internal Server Error, 503 during a restart): the message was fine, WordPress hiccupped, so the same send will succeed later. PUT is safely repeatable. Do not retry 4xx — those mean the message or credentials were wrong (400 invalid data, 401 bad password, 403 not allowed), so the same send fails the same way; log it and alert a human.
